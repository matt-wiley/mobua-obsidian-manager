# Canonical field options
_updated: 2026-07-13 | relates: [[emergent-schema]], [[frontmatter-type-inference]], [[markdown-source-of-truth]]_

**Decision:** A frontmatter field may declare a **fixed, persistent option list** stored in the vault at `<vault>/.obsidian-manager/schema.yaml`. `GET /folders/{folder}/schema` merges it with the values actually seen: `options = canonical + [detected not in canonical]` (canonical order preserved). Explicit config also **promotes** the field to `enum` even when the `_detect_enum` heuristic wouldn't fire.

**Why:** Enum options were derived purely from live data (`_detect_enum` in `api/folders.py` returns distinct in-use values). A status no longer used by any file vanished from the dropdown, so a **new** record could not be given a retired value like `03 - Todo` — `EnumField` only re-adds the *current* record's own value, and a new record has none. Options are config, not data; they must survive when unused. This is the first deliberate exception to the [[emergent-schema]] — schema is still emergent by default, but a field can opt into a declared option set. Config lives in the vault (not SQLite) to honor [[markdown-source-of-truth]]: the vault stays the one place that defines structure. The file is non-`.md`, so `watcher.py` ignores it and it never enters the index.

**Consequence:** New module `backend/obsidian_manager/schema_config.py` owns load/get/set/delete with atomic `.tmp`+`os.replace` writes (folder key normalized, trailing-slash-insensitive). Endpoints `GET/PUT/DELETE /folders/{folder}/field-options[/{field}]` manage the lists. Frontend: `FieldOptionsEditor.svelte` (opened from the `⋯` button on frontmatter `ColumnHeader`s) edits them; `schemaStore.reload()` refetches after a save. Explicit config overrides [[frontmatter-type-inference]]'s auto-`text`/`_detect_enum` result for that field only. Format is `folders: {<folder>: {<field>: {options: [...]}}}` — nested dict leaves room for future per-field config (type, color) without a format break.
